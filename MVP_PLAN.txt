Cut & Click — Fullständig MVP-Plan (10 Faser)
Projektstatus vid start: Backend har authController (register/login/getProfile), bookingController (getAvailableSlots/createBooking/getUserBookings), postController (getAllHaircuts/getAllItems/createHaircut) och authMiddleware (protect). Frontend har React/Vite/TypeScript med Startup-, Login-, Home-, BarberDetail-, StoreServices-, ProductDetail-, Booking-, BookingConfirmed-, History-, Favourites- och ProfilePage. Supabase-schema har tabellerna users, haircut_posts, item_posts, bookings, opening_hours och reviews.

Vad MVP:n ska uppnå
En färdig, deployad webbapp där:

Gäster kan bläddra bland barbershops, se tjänster, produkter och öppettider utan konto.
Kunder kan registrera sig, logga in, boka tider, se bokningshistorik och spara favoriter.
Barbers kan registrera sig med barber-rollen, lägga upp klipptjänster och produkter, sätta sina öppettider och hantera sina bokningar.
Ingen betalning sker online — produkter visas bara som information om vad som finns i salongen.
Fas 1 — Autentisering & Roller (Backend + Frontend)
Mål
Säkra upp hela auth-flödet: registrering med rollval (barber/customer), inloggning, utloggning, gäst-läge och skyddade routes. Det här är grunden som allt annat vilar på.

1.1 Databasändring — users-tabellen
Nuvarande schema har role TEXT DEFAULT 'customer'. Det räcker för MVP men vi behöver säkerställa att rollen verkligen sätts korrekt vid registrering och att barbers har en separat struktur för sin salongsinfo.

Lägg till i Supabase (SQL):

sql
-- Säkerställ att roll-fältet bara kan vara giltiga värden
ALTER TABLE users ADD CONSTRAINT users_role_check
  CHECK (role IN ('customer', 'barber'));
-- Barber-specifik profilinformation (en-till-en med users)
CREATE TABLE IF NOT EXISTS barber_profiles (
  user_id       UUID PRIMARY KEY REFERENCES users(id) ON DELETE CASCADE,
  salon_name    TEXT NOT NULL,
  salon_address TEXT NOT NULL,
  city          TEXT NOT NULL,
  phone         TEXT,
  bio           TEXT,
  cover_image   TEXT,
  rating        FLOAT DEFAULT 0,
  total_reviews INTEGER DEFAULT 0,
  created_at    TIMESTAMPTZ DEFAULT NOW()
);
Varför separat barber_profiles? Kunder behöver inte salongsnamn, adress etc. Att hålla det separerat håller users tabellen ren och gör queries enklare.

Det befintliga schemat har booking_time TIMESTAMPTZ men

ALTER TABLE bookings

-- Unik constraint för att förhindra dubbelbokningar
ALTER TABLE bookings

2.2 Komplettera opening_hours-tabellen
Nuvarande schema har separata kolumner per dag (monday, tuesday...) som TEXT. Vi byter till en flexiblare struktur med öppning/stängningstid per dag:

sql
-- Ny struktur för öppettider
DROP TABLE IF EXISTS opening_hours;
CREATE TABLE opening_hours (
  id          SERIAL PRIMARY KEY,
  user_id     UUID REFERENCES users(id) ON DELETE CASCADE NOT NULL,
  day_of_week TEXT NOT NULL,  -- 'monday', 'tuesday', etc.
  open_time   TIME,           -- NULL = stängd den dagen
  close_time  TIME,
  is_closed   BOOLEAN DEFAULT FALSE,
  UNIQUE(user_id, day_of_week)
);
Varför? Det är enklare att queray per dag, och det är tydligare om en dag är stängd (is_closed = true).

2.6 Backend — requireBarber middleware
javascript
// server/middleware/requireBarber.js
async function requireBarber(req, res, next) {
  // protect körs innan detta, så req.user finns
  const { data } = await supabase
    .from('users')
    .select('role')
    .eq('id', req.user.id)
    .single();
  if (!data || data.role !== 'barber') {
    return res.status(403).json({ error: 'Endast barbers har tillgång' });
  }
  next();
}

10.1 Säkerhet — Backend
Rate limiting: npm install express-rate-limit
  - /api/auth/*: max 10 requests/minute per IP (förhindrar brute-force)
  - /api/bookings: max 30 requests/minute
Input-validering: npm install express-validator
  - registerUser: validera email format, password length, required fields
  - createBooking: validera att booking_date är i framtiden
  - createReview: validera stars 1-5, max content length
CORS: Uppdatera app.js med specifik origin:
  app.use(cors({ origin: process.env.CLIENT_URL }));
Helmet: npm install helmet
  app.use(helmet());  // Sätter säkra HTTP-headers
10.2 Säkerhet — Supabase RLS (Row Level Security)
sql
-- Aktivera RLS på alla tabeller
ALTER TABLE users ENABLE ROW LEVEL SECURITY;
ALTER TABLE bookings ENABLE ROW LEVEL SECURITY;
ALTER TABLE haircut_posts ENABLE ROW LEVEL SECURITY;
ALTER TABLE item_posts ENABLE ROW LEVEL SECURITY;
ALTER TABLE favourites ENABLE ROW LEVEL SECURITY;
ALTER TABLE reviews ENABLE ROW LEVEL SECURITY;
ALTER TABLE notifications ENABLE ROW LEVEL SECURITY;
-- Exempel policies:
-- Kunder kan bara se sina egna bokningar
CREATE POLICY "Users see own bookings"
  ON bookings FOR SELECT
  USING (auth.uid() = customer_id);
-- Barbers kan se bokningar för sina egna tjänster
CREATE POLICY "Barbers see their bookings"
  ON bookings FOR SELECT
  USING (
    EXISTS (
      SELECT 1 FROM haircut_posts
      WHERE id = bookings.haircut_id AND user_id = auth.uid()
    )
  );
-- Alla kan läsa haircut_posts och item_posts
CREATE POLICY "Public read haircut_posts"
  ON haircut_posts FOR SELECT USING (true);
